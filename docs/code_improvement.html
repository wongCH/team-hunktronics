Lead developer assessment

Make change cheap, not architecture impressive.

The repository has a sound process boundary, strict TypeScript, fast co-located tests, and small provider adapters. Its maintenance risk is concentrated in state ownership: persistence updates can race, run completion has several owners, and the renderer carries duplicate run controllers.

Recommendation: fix the behavioral boundaries first, delete duplicate paths second, and move files only when a tested feature slice is already changing.

Verdict

Keep the skeleton. Repair ownership.

This is not a repository that needs a clean-room reorganization. The existing main, preload, renderer, and shared boundaries are correct for Electron. XP value comes from shorter feedback and one owner per behavior.

B-

Structurally healthy, operationally uneven

The code is testable and understandable in slices, but central files mix responsibilities and several asynchronous projections rely on timing rather than completion. Those are maintenance and correctness problems, not merely style concerns.

Prioritized findings

Fix behavior before moving folders

Each recommendation includes the cheapest check that can disprove the proposed fix. That is the XP contract: no non-trivial change without a fast executable signal.

Critical

Serialize the whole persistence mutation

writeJson queues writes, but several collections read before joining that queue. Concurrent task, schedule, pipeline, artifact, action, approval, skill, connection, or settings saves can write stale snapshots over each other. Vault writes also need explicit corruption handling and atomic serialized replacement.

Evidence: src/main/store.ts, especially writeJson, updateJson, saveTask, and sibling save methods; src/main/vault.ts.
Smallest checkStart two deferred saves to distinct records, resolve together, and assert both survive. Parameterize across collection methods.
Critical

Route delegation using the persisted capability manifest

Agent summaries deliberately omit full SOUL content, but automatic delegation and delegation prompts still inspect candidate.soul. Production routing can therefore ignore the capability data that persistence created, while unit fakes with hydrated agents continue to pass.

Evidence: src/main/store.ts toAgentShell/listAgents; src/main/runService.ts delegationInstructions/automaticDelegation.
Smallest checkUse a real temporary Store, save a capability-only report, run routing, and assert the expected direct report is selected.
High

Give terminal run effects one awaited owner

Task, schedule, pipeline, and memory projections are launched independently. The renderer then reloads after arbitrary 50, 80, or 100 ms delays. Slow storage can return stale state even though the run event has already arrived.

Evidence: src/main/index.ts run subscription; src/main/ipc.ts task projection; TasksPage.tsx; SchedulePanel.tsx; PipelinePanel.tsx.
Smallest checkHold persistence with a deferred promise and assert no projection event is published until the save resolves.
High

Delete duplicate run controllers and the unused raw-chat path

AgentRunner and useChatStore independently own conversation identity, event correlation, streaming, refresh, start, and cancel. Main IPC also carries a separate raw provider chat path with no renderer caller. Reuse the canonical chat store and remove code before inventing a shared controller.

Evidence: AgentRunner.tsx; useChatStore.ts; raw chat block in src/main/ipc.ts; no renderer use of api.chat.send.
Smallest checkOpen the same agent twice, send once, and assert one direct conversation plus one runs.start call.
High

Validate every renderer mutation at the main-process boundary

Several writes are validated, but conversation saves, settings patches, and tool authorization accept typed renderer values as if TypeScript were runtime validation. Tool side-effect class should be derived in main from a tool allow-list before execution exists.

Evidence: conversation, tool-policy, and settings handlers in src/main/ipc.ts; approval behavior in src/main/toolPolicy.ts.
Smallest checkInvoke captured IPC handlers with malformed payloads and assert rejection without persistence.
Medium

Preserve edits during agent SOUL hydration

The agent list returns a shell and the editor later hydrates SOUL content. The hydration callback replaces the entire draft, so a fast edit to name, role, model, or another enabled field can disappear.

Evidence: hydration effect in AgentEditor.tsx; shell/hydrated split in src/main/store.ts.
Smallest checkDelay agents.get, edit Name, resolve hydration, and assert Name remains while SOUL is populated.
Medium

Make startup failure recoverable

App fire-and-forgets initializers while loadAll only sets ready after an all-or-nothing Promise.all. One rejected IPC call can leave the application on Loading indefinitely.

Evidence: App.tsx; useAppStore.ts; concurrent agent initialization consumers.
Smallest checkReject startup once, retry successfully, and assert state becomes ready. Assert concurrent init calls make one IPC request.
Medium

Constrain file reads and completed-run retention

Managed memory confines paths lexically but can follow a Markdown symlink outside its root. Separately, completed idempotency and response promises are retained for the app lifetime. Reuse the existing LLM-wiki realpath pattern and bound completed-run history.

Evidence: walk/readPath in memoryService.ts; confinedFile in llmWikiService.ts; maps in runService.ts.
Smallest checkReject an external symlink; complete more than the retention limit and assert old entries are evicted without breaking active idempotency.
Medium

Add one non-writing verification command

The repository has typecheck and tests, but no canonical gate. Packaging can proceed without either. Prettier is write-only and the existing tree is not yet at a clean check baseline.

Evidence: scripts in package.json; test config in vitest.config.ts; scripts table in README.md.
Smallest checkAfter one isolated formatting baseline, run npm run check locally and in packaging.
XP plus Ponytail

The rules for every improvement slice

XP supplies the feedback discipline. Ponytail supplies the complexity budget. Together they favor a small tested correction over a broad architectural program.

01 / TEST FIRST

Expose the failure

Add one focused check that fails for the current behavior. Avoid snapshot suites and broad fixtures.

02 / ONE OWNER

Fix the controlling path

Repair the service where all callers route, not each renderer symptom or timer.

03 / DELETE FIRST

Remove duplicate paths

Reuse useChatStore and trusted runs. Do not build another run-controller abstraction.

04 / SMALL RELEASE

One behavior per change

Keep persistence, routing, runtime events, and folder moves in separate reviewable changes.

05 / MOVE ON TOUCH

Rehome only active files

Move a cohesive feature cluster when it receives a real behavior change, then fix imports mechanically.

06 / YAGNI GATE

Demand evidence for tools

No database, router, DI container, schema framework, or monorepo until a measured constraint requires it.

Ponytail guardrails

Keep, change, avoid

The target is not maximum modularity. It is minimum cognitive load with reliable behavior. These constraints prevent the cleanup from becoming its own maintenance problem.

Keep

  • Electron process boundaries
  • Strict TypeScript config
  • Co-located Vitest files
  • Dependency-record test seams
  • Provider registry and adapters
  • Simple in-memory page switch
  • One-line renderer API seam
  • JSON storage at current scale

Change when touched

  • Queued storage mutations
  • Runtime event ownership
  • Delegation policy extraction
  • Renderer feature locality
  • Startup and IPC error states
  • Documentation drift
  • Unused tracked assets
  • Manual diagnostic packaging

Do not add

  • DI container
  • Repository interfaces per entity
  • Generic CRUD hooks
  • One-component folders
  • Barrel files everywhere
  • SQLite without measured limits
  • Router without deep links
  • Monorepo or package split
