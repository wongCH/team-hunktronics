1 Introduction

This document captures the functional and non-functional requirements and the technical design of the AI Agent Control Panel (internally agent-control-panel). It is intended for engineers, reviewers, and QA working on the product.

Purpose

Provide a single, secure desktop client that lets a user configure connections to several Large Language Model (LLM) providers, store credentials safely, and hold streaming chat conversations that persist locally.

Product perspective

The application is a self-contained Electron desktop app. It has no backend server of its own; each configured connection talks directly from the user's machine to the chosen provider's HTTPS API. All state (connections, conversations, settings, encrypted secrets) is persisted to the OS user-data directory.

1.1 Scope

In scope

  • Managing multiple provider connections (Ollama, OpenAI, Anthropic, GitHub Models, OpenAI-compatible endpoints, GitHub Copilot).
  • Encrypted, OS-backed storage of API keys and tokens.
  • GitHub OAuth device-flow login for token-based providers.
  • Model discovery, streaming chat with cancellation, and locally persisted conversations.
  • Application settings including active connection/model and experimental toggles.

Out of scope (MVP)

See §12 Out of Scope.

2 System Overview

The product follows Electron's three-process security model. The React renderer never touches Node APIs directly; it communicates with the main process exclusively through a small, allow-listed, typed IPC surface exposed by the preload bridge. Secrets are held, encrypted, and used only within the main process.

3 Architecture

3.1 Process responsibilities

  • Main — window lifecycle, security policy (navigation/window-open handling), IPC registration, provider orchestration, encrypted vault, plaintext store, and GitHub device flow.
  • Preload — exposes window.api with namespaced methods (vault, connections, secrets, models, chat, conversations, settings, github, shell). Only these channels are reachable from the UI.
  • Renderer — React UI plus two Zustand stores: useAppStore (connections, settings, models, modals, vault status) and useChatStore (conversations, streaming lifecycle).

3.2 Streaming chat flow

  1. Renderer calls chat.send(); main validates the connection/model, resolves the provider and context, and creates a streamId plus an AbortController.
  2. The provider streams tokens; the main process forwards each delta to the renderer via chat:chunk events, then emits chat:done or chat:error.
  3. The renderer appends deltas to the trailing assistant message and persists the conversation on completion.
  4. chat.cancel(streamId) aborts the underlying request; the stream is removed from the active map.

3.3 Persistence layout

8 Non-Functional Requirements

8.1 Performance

IDRequirement
NFR-PERF-01Assistant responses must render incrementally as tokens arrive; the UI must not block during generation.
NFR-PERF-02Generation runs on background tasks in the main process; multiple streams may run concurrently.
NFR-PERF-03The window must appear without a blank flash (deferred show until ready).
NFR-PERF-04Vault contents are cached in memory after first read to avoid repeated disk/decrypt cost.

8.2 Reliability

IDRequirement
NFR-REL-01Provider and IPC errors are surfaced as user-facing messages rather than crashing the app.
NFR-REL-02Missing or corrupt persistence files fall back to safe defaults (empty lists / default settings).
NFR-REL-03Device flow tolerates pending/slow-down responses and honours expiry and denial.
NFR-REL-04Aborted or completed streams are removed from the active-stream registry to prevent leaks.

8.3 Usability & UX

IDRequirement
NFR-UX-01Native-feeling macOS chrome (hidden-inset title bar, custom traffic-light offset, dark-mode support).
NFR-UX-02Connection testing provides clear success/failure feedback including model counts.
NFR-UX-03Model selection offers fallback suggestions when a provider has no list endpoint.
NFR-UX-04Rich Markdown chat rendering with code syntax highlighting.

8.4 Portability & Compatibility

IDRequirement
NFR-CMP-01Primary target is macOS arm64, packaged as DMG and ZIP.
NFR-CMP-02Secret storage abstracts across macOS Keychain, Windows DPAPI, and Linux secret service, keeping the codebase cross-platform ready.
NFR-CMP-03Requires Node.js 18+ and runs on Electron 31 / Chromium.

8.5 Maintainability

IDRequirement
NFR-MNT-01TypeScript is used across all processes with a single shared types module for cross-process contracts.
NFR-MNT-02Providers implement a common Provider interface, so new backends can be added without touching the UI or IPC layer.
NFR-MNT-03IPC channel names are centralized as constants to avoid string drift.
NFR-MNT-04Code is formatted with Prettier and type-checked via a dedicated script.

8.6 Privacy & Observability

IDRequirement
NFR-PRV-01Local-first: no telemetry or analytics; all data stays in the user-data directory.
NFR-PRV-02Requests go directly from the user's machine to the selected provider; no intermediary server.
NFR-OBS-01Detached DevTools are available in development; runtime failures are returned through IPC responses/events.

9 Security Requirements

Security is a first-class, non-negotiable concern for this application.

IDRequirementPriority
SEC-01The renderer runs with contextIsolation: true, nodeIntegration: false, and sandbox: true.Must
SEC-02Secrets are encrypted at rest with the OS keychain (safeStorage), stored as base64 ciphertext with file permission 0600.Must
SEC-03Plaintext secrets never cross IPC to the renderer; the secrets API is write-/presence-only.Must
SEC-04Only an explicit allow-list of IPC channels is exposed to the UI through contextBridge.Must
SEC-05In-app navigation is blocked; only http(s) links are opened externally, and window.open is denied.Must
SEC-06Provider type and connection payloads are validated against an allow-list before use.Must
SEC-07The openExternal handler validates the URL scheme to prevent arbitrary command/URI execution.Must
SEC-08Encryption must fail closed: if OS encryption is unavailable, storing/decrypting secrets throws rather than persisting plaintext.Must
SEC-09No secrets, tokens, or credential files are committed to the repository; local data files are git-ignored, and only a public OAuth client id is embedded.Must
SEC-10The experimental Copilot path is disabled by default and gated behind an explicit user opt-in.Should
OWASP alignment: The design mitigates common risks — secrets are never exposed to the untrusted renderer (sensitive-data exposure), IPC input is validated (injection), navigation and link handling are constrained (SSRF/redirect), and the client stores no secrets in source control.

11 Constraints & Assumptions

Constraints

  • Must run as a desktop app; no server-side component is permitted.
  • Secrets must be handled only in the main process and encrypted by the OS keychain.
  • The renderer must remain sandboxed with context isolation on.
  • OS encryption must be available for secret-backed providers to function.

Assumptions

  • The user supplies valid provider credentials or a reachable local endpoint (e.g. Ollama).
  • Network access to the selected provider's HTTPS API is available.
  • The embedded GitHub OAuth client id is a public client identifier (not a secret).
