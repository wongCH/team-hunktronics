1 Product overview

The application is a desktop-only Electron system with no application-owned backend service. The renderer controls configuration and presents state; the sandboxed preload bridge exposes an allow-listed API; the main process owns credentials, provider traffic, trusted context assembly, run state, persistence, scheduling, and policy decisions.

Conversation

Persistent direct and delegated threads, Markdown output, streaming, cancellation, new sessions, clear-chat behavior, and per-agent chat.

Agent team

One orchestrator, optional team leads, and specialists. The team graph powers automatic domain routing and nested delegation.

Managed work

Tasks, sequential pipelines, app-open schedules, memory review, approvals, traces, and bounded local-data inspection.

1.1 In-scope capabilities

  • Seven provider adapters and per-connection model discovery.
  • OS-backed encrypted secrets that never return to the renderer.
  • Main-owned trusted runs with persisted conversations and live run events.
  • Hierarchical agent editing, team visualization, archiving, tools, skills, and autonomy metadata.
  • Specialist-first automatic routing with team-lead and orchestrator fallbacks.
  • Managed Markdown memory plus read-only human LLM-wiki context.
  • Durable tasks, one-to-eight-stage pipelines, cron schedules, policy records, and API traces.
  • Six persisted visual themes; Indigo Nebula is the default.

2 Architecture

Renderer
React 18 pages and components; Zustand stores for application, chat, and agents; React Flow plus Dagre for the team graph. No direct Node.js access.
Preload
Sandboxed contextBridge exposing 20 namespaced APIs and callback subscriptions. It contains no business logic and returns unsubscribe functions for events.

IPC validation and orchestration, RunService, context assembly, Store, Vault, memory, LLM wiki, schedules, pipelines, tool policy, provider adapters, and GitHub device flow.
External systems
Selected LLM endpoints, GitHub OAuth endpoints, the OS keychain, local JSON/Markdown files, and the OS browser for approved http(s) links.

3 Trusted run lifecycle

Regular chat and agent work use runs:start, not the lower-level chat:send transport. The main process owns history, agent identity, context, persistence, and lifecycle transitions.

1. ValidateConversation, input, idempotency key, agent.
2. ResolveAgent or default connection/model.
3. AssembleSoul, policy, wiki, skills, memory, history.
4. ExecuteProvider stream or delegated child runs.
5. PersistAssistant result, terminal state, trace, daily log.

3.1 Context ordering and budget

  1. Agent soul identity, when an agent is selected.
  2. Runtime authorization and delegation transport instructions.
  3. Bounded, read-only human LLM-wiki context.
  4. Assigned skill instructions in configured order.
  5. Team baseline memory and agent baseline memory.
  6. Newest conversation history that fits.
  7. The current user input.

Default context window

32,768 estimated tokens.

Completion reserve

4,096 estimated tokens.

Estimator

Approximately four characters per token plus per-message overhead.

Required identity, runtime instructions, wiki, skills, memory, and current input fail closed if they exceed the budget. Older history is dropped newest-first only after required context fits.

3.2 State and idempotency

  • Run states: queued, running, completed, failed, cancelled.
  • Duplicate idempotency keys return the existing run view/completion.
  • Active runs and completion promises are process-local and disappear on restart.
  • Conversation messages, provider traces, task links, schedule state, pipeline state, and artifacts are durable.
  • Parent cancellation recursively aborts active delegated children.

3.3 Lower-level direct chat path

chat:send remains a one-shot provider transport with streaming events and traces. It is used by AI-assisted persona generation. It does not load authoritative conversations, managed memory, skills, or team routing.

4 Team model and delegation routing

4.1 Hierarchy invariants

Required graph rules

  • Exactly one active orchestrator when the team is non-empty.
  • The orchestrator has no reporting manager.
  • Every other active agent reports to an active manager.
  • Specialists cannot manage agents.
  • No cycles; maximum graph depth is three tiers.
  • Specialists cannot receive the code tool grant.

Relationship storage

  • reportsTo is the canonical manager edge.
  • delegatesTo is derived from canonical child edges.
  • Agent save/delete normalizes and validates the complete graph.
  • Agent mutations are queued atomically so concurrent child creation preserves every registration.

4.2 New-agent creation flow

Creating a team or adding a team member crosses the renderer/preload trust boundary and commits the complete team graph as one validated mutation.

1. Capture The modal requires a name and, for non-orchestrators, an eligible reporting manager. It may build or generate the agent soul.
2. Construct useAgentStore assigns the ID, role defaults, canonical reportsTo, draft autonomy, timestamps, and empty grants.
3. Validate IPC agents:save trims and bounds the payload, accepts only known roles and autonomy levels, and discards incoming derived delegatesTo.
4. Register + persist A queued read-modify-write inserts the agent, derives every manager's delegatesTo, validates the full hierarchy, then atomically writes agents.json.
5. Complete The normalized roster returns to the renderer, the new agent is selected, and onCreated advances the UI to that agent. No per-agent folder is created in this transaction.
Failure path. Invalid names, managers, roles, cycles, depth, tool grants, or storage operations reject the save. No invalid graph is committed; the creation modal stays available and displays the propagated error.

Filesystem and dependent artifacts

Not an execution workspace. The current implementation has no per-agent checkout, working directory, sandbox, or arbitrary file workspace. The physical path <userData>/memory/agents/<agent-id>/ is a managed Markdown memory scope and is created lazily, not when the agent record is saved. On macOS, <userData> is typically ~/Library/Application Support/Agent Control Panel.

4.3 Routing priority

Execution order: matching specialist first; matching team lead if no specialist matches; the orchestrator performs the work only when neither tier matches.
  1. The manager scores active direct reports and specialists under each direct-report branch using names, titles, skill IDs, and bounded soul capability text.
  2. Explicit agent-name matches receive the highest score. Specialists qualify with one meaningful lexical overlap; team leads require two.
  3. If a nested specialist matches, the orchestrator delegates to that specialist's direct team-lead branch; the team lead then routes to the specialist.
  4. Multiple equally relevant branches may fan out, capped at three concurrent direct reports.
  5. Each parent synthesizes child results into its own output. Child failures are supplied as synthesis data rather than silently converted into success.

4.4 Delegation safety limits

6.2 JSON reliability contract

  • Missing files return typed defaults.
  • Non-vault writes are serialized per file.
  • Writes use a mode-0600 temporary file, file sync, and atomic rename.
  • A valid current file is copied to .bak before replacement.
  • A malformed primary file recovers from its valid backup.
  • If both primary and backup are invalid, the read fails explicitly rather than returning a success-shaped empty value.

6.3 Managed memory tree

memory/
  team/
    MEMORY.md
    <evergreen>.md
  agents/<agent-id>/
    MEMORY.md
    YYYY-MM-DD.md
    <evergreen>.md
    archive/
  .history/
  • Only team/MEMORY.md is initialized automatically when memory is first accessed. Agent directories and documents are lazy first-use artifacts.
  • Safe Markdown names and path confinement prevent writes outside the managed root.
  • Documents are limited to 100 KB; baseline files are limited to 200 lines.
  • Optimistic revision checks prevent overwriting a concurrently changed document.
  • Previous versions are retained under .history.
  • Terminal agent runs append dated per-agent daily log entries.
  • Search is local lexical scoring; health checks detect size, age, duplicates, missing links, and baseline limits.
  • Compression is propose-then-apply; accepted source daily logs move into archive.

6.4 Human-owned LLM wiki

A configured external wiki is treated as bounded, read-only run context. The app can scaffold an initial LLM-Vault, select an existing vault, or remove the reference. During runs it reads identity, schema/rules, selected index sections, the latest action page, and the recent log tail.

  • Required root files: identity.md, index.md, and one of SCHEMA.md, AGENTS.md, or CLAUDE.md.
  • Symlinked roots/files and paths resolving outside the vault are rejected.
  • Individual source files must be regular files no larger than 1 MB.
  • Scanning stops after depth eight or 2,000 Markdown files and excludes raw/archive/hidden trees.
  • The general settings API cannot directly change llmWikiPath; dedicated handlers own that reference.

9.3 Conversation behavior

  • New session persists a blank, unbound conversation and selects it.
  • Clear chat preserves agent/provider binding while removing messages; direct-agent titles remain stable.
  • Direct agent conversations are reused; delegated threads are excluded from direct-thread lookup.
  • Session mutations and destructive controls are disabled during streaming.
  • Terminal run events reload authoritative conversations from main-process persistence.

10 Work automation and operations

10.1 Tasks

  • Tasks move through backlog, in-progress, review, and done.
  • Starting a task requires an assigned, configured agent and creates/reuses a conversation.
  • The trusted run receives a review-ready task prompt.
  • Successful terminal runs move the task to review; failures retain status and record the error.

10.2 Pipelines

  • An orchestrator or team lead owns one to eight sequential stages.
  • Every stage agent must be an active direct report of the owner.
  • Each stage requires a name, instructions, and expected-output contract.
  • The service persists a stage brief, runs the agent, versions the output, and feeds prior outputs into the next brief.
  • A completed final stage moves the execution to review; failure/cancellation stops progression while preserving existing artifacts.

10.3 Schedules

  • Cron expressions are evaluated with an IANA timezone through cron-parser.
  • The main process polls every 30 seconds while the app is open and supports manual run-now.
  • Each schedule reuses a dedicated conversation and deterministic occurrence idempotency key.
  • Definitions persist maxAttempts, but no automatic retry loop currently consumes it.
  • There is no OS daemon; schedules do not run while the desktop application is closed.

10.4 Policy, traces, and local data

  • Tool policy checks grants and side-effect class, then approves, denies, or creates a 15-minute approval request.
  • Sensitive argument keys are redacted; a SHA-256 digest represents the original argument object.
  • The policy broker records authorization only; it does not execute tools.
  • Operations receives live trace updates and reads durable provider diagnostics.
  • Local Data searches up to an allow-listed 500-row result; the encrypted vault and managed memory tree are excluded.

11 Security and privacy

Electron boundary

  • contextIsolation: true
  • nodeIntegration: false
  • sandbox: true
  • Blocked in-app navigation and window creation
  • Only http(s) URLs open externally

Secret boundary

  • Electron safeStorage encrypts secrets.
  • Plaintext stays in the main process.
  • Renderer APIs are set/clear/presence only.
  • Secret writes fail closed without OS encryption.
  • Vault file mode is 0600.

Content boundary

  • Main-owned context and conversation persistence.
  • Bounded context and delegated payloads.
  • Confined memory and LLM-wiki paths.
  • Skill import type/size checks.
  • Team graph validation before persistence.

Audit boundary

  • Persisted provider status/counts and response preview.
  • No raw prompt or injected memory fields in trace schema.
  • Sanitized tool arguments and original digest.
  • Explicit approval decisions and expiry.
  • Read-only allow-listed data explorer.

11.1 Privacy statement

There is no application telemetry or app-managed cloud synchronization. Configuration and operational state are local. Prompts and responses necessarily transit the selected provider. A bounded model-generated response preview is retained locally in traces for diagnostics.

Known vault reliability caveat. Unlike the non-secret store, the vault currently writes directly and treats any read/parse failure as an empty cache. It does not use the JSON store's atomic backup/recovery path.
